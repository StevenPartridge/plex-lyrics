Lyrics Companion Overhaul

Planning UX First

Task plan for transforming the extension from a timer-driven auto-clicker into a trustworthy lyrics companion for side-screen Plex music listening.

Scope

The outcome stays the same: a Plex listener can keep music lyrics visible with almost no effort. Everything else is negotiable - timer, layout, controls, wording, and implementation model.

AreaCurrent AppTarget Experience
Product job Repeatedly click the Plex lyrics button for a selected duration. Keep lyrics available while respecting user intent, especially when the user manually closes lyrics.
Mental model A timer is running in the background. A companion is either keeping lyrics open, paused by you, or off.
Primary control Start and Stop buttons with duration radio choices. A clear session toggle plus one-click Resume, Pause, and Stop actions based on current state.
Manual close behavior The interval can reopen lyrics shortly after the user closes them. Manual close immediately pauses/cancels automation and shows visible confirmation in the page, popup, and toolbar badge.
Status visibility Status mostly exists inside the popup, and the popup is usually closed. The toolbar badge and optional in-Plex toast make the state visible without hunting through the extension UI.
Timer role Timer is the main product surface. Timer becomes optional guardrail copy, not the core interaction. The product centers on user intent and session state.
Extension platform Firefox Manifest V2 with an event background script and timer-based interval behavior. Manifest V3 with cross-browser background registration, storage-backed state, alarms or rehydration where needed, and observer-first content behavior.
Non-goals Do not build a general Plex skin, lyric source scraper, account system, or landing page. This stays a small focused WebExtension.
North star: the extension should feel like it is on the listener's side. If they close lyrics, it should treat that as a command, not a problem to undo three seconds later.

Product Principles

PrincipleDesign Consequence
Trust user intentManual close always wins. No automatic reopen until the user explicitly resumes.
Make state obviousUse concise status language: Keeping lyrics open, Paused by you, Off, Waiting for Plex.
Reduce decisionsDefault to a session-based mode. Move durations into an optional secondary control if they survive testing.
Stay out of the musicIn-Plex UI should be small, calm, and temporary unless the user chooses a persistent indicator.
Recover quicklyEvery paused/error state gets one obvious action: Resume lyrics, Open Plex, or Try again.

Key Decisions

Decisions Q1-Q5 were locked on 2026-06-06 from wiki feedback. The visual direction is Plex-native integration: the final UI should feel like a missing Plex Music feature, not a generic browser-extension panel.

DecisionChoiceImpact
Manual close Pause/cancel automation Closing lyrics by hand enters pausedByManualClose; automation does not reopen lyrics until Resume.
Visual confirmation Visible outside popup Use browser action badge and a short in-Plex confirmation so the cancellation is not hidden.
Timer prominence Demote timer The core UI is state-based. Time limits can remain as an advanced session option if useful.
Interface direction Control center The popup becomes a compact status-and-action surface, not a form full of duration choices.
Q1 - Session boundary A - Until stopped or Plex closes Default action is Keep lyrics open. No timer-first UI; optional limits can be revisited later.
Q2 - Status surface A - Toast plus toolbar badge Manual close and other important events get a temporary in-Plex toast plus persistent browser action badge state.
Q3 - Extension platform B - Migrate to MV3 now The overhaul includes manifest, background lifecycle, alarm/storage, and MV3-safe messaging changes. Firefox uses MV3 background scripts; browsers that support service workers use the same file as a service worker.
Q4 - Opening trigger A - Observer first, fallback poll Content script observes Plex/lyrics state first, then uses a slower guarded fallback poll only when observation misses.
Q5 - Visual ambition A - Plex-native integration Popup, toast, and badge design should follow Plex's dark, compact, gold-accented design language closely enough to feel native.
Q1
What should the default session boundary be?

This determines the primary label in the redesigned popup and how much timer UI survives. It blocks the session-state implementation and final popup layout.

Q2
How visible should the in-Plex status be?

This blocks the content-script UI. The product needs visible confirmation, but Plex should still feel like Plex.

Q3
Should this overhaul include a Manifest V3 migration?

The UI/state work can ship on Firefox Manifest V2. Manifest V3 may be worth doing eventually, but it changes background lifetime assumptions and broadens the project.

Q4
How should the extension decide when to open lyrics?

The current three-second interval is easy to reason about but creates surprise. This decision blocks the content/background refactor.

Q5
What visual direction should the overhaul pursue?

This blocks popup, toast, badge, and screenshot implementation. The current mockups were too generic; the final direction should either merge into Plex's design language or make a deliberate, high-polish companion statement.

Step Plan

Implementation should be product-led: first make the state machine impossible to misunderstand, build the MV3 foundation around that state, then lock the visual direction before any interface build-out.

Phase 1 - Define the event/state contract

Write the durable states and events before UI work: idle, active, pausedByManualClose, pausedByUser, waitingForPlex, and lyricsUnavailable. Events include start, stop, resume, manualCloseDetected, lyricsOpened, lyricsMissing, and plexUnavailable.

Phase 2 - Migrate the extension foundation to MV3

Update manifest.json to Manifest V3, move browser_action to action, split host permissions from extension permissions, and register the background file for both Firefox MV3 background scripts and service-worker browsers. Rebuild runtime assumptions around storage, messages, and alarms instead of long-lived setInterval and setTimeout.

Phase 3 - Build manual-close detection and stop the surprise loop

Teach content.js to report visible-state transitions. Synthetic clicks set a short suppression window; a user-driven transition from visible to hidden enters pausedByManualClose. Background clears automation immediately and stores the paused reason.

Phase 4 - Replace timer-first storage with MV3-safe session state

Replace enabled, duration, and endTime as the primary model with a single stored session object: state, reason, active tab IDs, last event, last synthetic click time, and MV3 background rehydration metadata. With Q1 locked to until-stopped, omit timer controls from the first redesigned surface.

Phase 5 - Lock visual direction with reviewed mockups

Q5 is locked to Plex-native integration. Use the Plex-native mockups as the source of truth, then produce enough final visual detail to prevent the implementation from drifting into generic extension UI: active popup, paused popup, in-Plex toast, toolbar badge states, and at least one narrow/mobile popup crop.

Phase 6 - Redesign popup as a polished control center

Build the popup around state: headline, one-sentence explanation, primary action, secondary action, current Plex detection, and last event. Remove the stacked radio list and timer display from the default surface. The visual system must follow Q5, not generic browser-extension defaults.

Phase 7 - Add toast and toolbar badge status

Use the browser action badge for persistent state and the chosen in-Plex surface from Q2 for important moments. Manual close shows concise copy like Lyrics paused, then leaves the badge in a paused state.

Phase 8 - Make opening smarter and quieter

Implement the Q4 trigger strategy: observe Plex/lyrics DOM changes in the content script, click only in active, return explicit content statuses, and keep a slower guarded fallback poll for missed transitions or background wakeups.

Phase 9 - Polish, test, and ship

Run manual extension smoke tests across active, paused, stopped, no-Plex, multiple-tabs, track-change, unavailable-lyrics, popup-reopen, browser-restart, and MV3 background wakeup cases. Update screenshots and release notes only after behavior feels calm under repeated use.

Experience Blueprint

The redesign should feel modern and fun without becoming noisy. The popup is a tool surface, not marketing. It should be dense enough for repeated use and warm enough that status changes feel human.

Popup Structure

Primary action Keep lyrics open, Resume lyrics, or Open Plex, depending on state. One obvious next action; no Start/Stop ambiguity.
Session controls Small secondary controls: Pause, Stop, optional Stop after.... Power without clutter. Durations are not the page's personality anymore.
Plex detection Current tab status, lyrics visibility, and whether the button was found. When something does not work, the popup should explain why without logs.

Visual Tone

Use a dark, Plex-adjacent interface with one lively accent, crisp status pills, and button labels that name real actions. Avoid a giant hero, decorative cards, or a timer-dominated dashboard.

Manual Close Moment

When the user closes lyrics, show a temporary in-Plex confirmation and switch the badge to paused. The next popup open should lead with the same reason, not a generic stopped state.

State Model

The product gets reliable only when the background script owns a real state machine and every surface renders from it. Popup, badge, and content script should not each invent their own truth.

StateEnter WhenAutomation BehaviorExit
idle User has not started, stopped explicitly, or session ended. No clicking. Badge says Off or clears. Start from popup.
active User chooses Keep lyrics open or resumes after a pause. Open lyrics when Plex has a lyrics button and lyrics are not visible. Manual close, Pause, Stop, Plex unavailable, or optional time limit.
pausedByManualClose Lyrics were visible and become hidden from a user action, not an extension click. No clicking. Badge says Pause. Popup leads with the reason. Resume or Stop.
pausedByUser User pauses from popup or badge action. No clicking. Preserve context. Resume or Stop.
waitingForPlex No matching Plex tab, Plex tab not ready, or button not found yet. Do not spam clicks. Watch or poll slowly. Plex detected or Stop.
lyricsUnavailable Plex indicates no lyrics button or no lyrics for the current track. Stay active but quiet. Recheck on track change. Track change, Stop, or manual pause.
Implementation detail: content responses should become explicit statuses such as clicked, already-visible, manual-close-detected, button-missing, and lyrics-unavailable. The current unconditional clicked response hides useful product state.

This is a trust feature, so QA should focus on repeated real-world sequences rather than only happy-path clicks.
