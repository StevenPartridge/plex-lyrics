Lyrics PiP Research & Build Plan

Research UX Heavy Release Risk Review

Task plan for turning Plex's currently available lyrics into a beautiful always-on-top Picture-in-Picture lyric companion without fetching, storing, or redistributing lyrics from any other source.

Scope

The dream is not a lyric scraper and not a replacement lyrics service. It is a local mirror of the lyric experience Plex already shows to the user: the same available lyrics, the same timed-line behavior when Plex exposes it, rendered in a dedicated always-on-top window for side-screen listening.

AreaPlanReason
Product promise Open a focused PiP lyrics view while Plex Music keeps playing in the original tab. Solves the "lyrics disappear when I navigate away" problem without changing the music source.
Lyrics source Only mirror lyrics that Plex Web already displays or exposes in the current page. Keeps the product inside the user's existing Plex/LyricFind entitlement instead of becoming a lyric distribution app.
Sync behavior Observe Plex's highlighted current line if available; fall back to static lyric display if Plex only has plain lyrics. Plex officially supports both timed and plain lyrics, so the UI must handle both gracefully.
Primary runtime Document Picture-in-Picture for arbitrary HTML; classic video PiP only as a possible fallback. Lyrics need real layout, text, resizing, and accessibility. Classic PiP is video-only.
Non-goals No lyric API scraping, no third-party lyric lookup, no saving full lyric bodies, no public lyric sharing, no Plex Pass bypass, no fake affiliation with Plex, LyricFind, or Musixmatch.
North star: make Plex Music feel like it gained an elegant floating lyric stage. The extension should be invisible as a data system and unmistakable as a local viewing aid.

Research Snapshot

Verdict: technically plausible, worth spiking, and potentially publishable if the release posture stays conservative. The riskiest areas are not PiP itself; they are lyrics rights, Plex terms, naming/branding, and browser-specific behavior in Firefox/Zen.

QuestionFindingImplication
Can arbitrary lyrics HTML live in PiP? Yes where Document Picture-in-Picture exists. MDN describes an always-on-top window populated with arbitrary HTML, unlike video-only PiP. Build the real experience in HTML first, with text semantics and responsive layout.
Can the popup open it directly? Maybe, but the API requires transient user activation in the page context. The robust path is an injected "Lyrics PiP" control inside Plex Web. The extension popup can focus Plex and guide the click.
Can Plex provide enough lyric state? Plex's support docs say Plex Web supports LyricFind lyrics and timed lyrics highlight the current line as the song progresses. First spike must map the actual DOM for line text, active-line class, track changes, and no-lyrics states.
Is this a clean store submission? It can be if it is single-purpose, transparent, local-only, permission-minimal, and non-infringing. Ship with a privacy policy, no analytics by default, no remote code, and clear "not affiliated" wording.
Is this clean under Plex terms? Not guaranteed. Plex acknowledges third-party "Interfacing Software" but restricts modification, derivative use, content copying, and unauthorized sharing. Personal/local mirror is the lowest-risk interpretation. Public release needs a preflight legal/terms review and possibly Plex contact.
Release gate: do not market this as a lyric provider or Plex replacement. Do not persist or export lyric text. Do not use Plex logos or imply endorsement. This page is product/legal risk analysis, not legal advice.

Live Spike Findings

Live Plex Web inspection on 2026-06-13 proved the timed-lyrics DOM is mirrorable without inventing lyric timing. Findings below intentionally avoid storing lyric text; the spike recorded selectors, classes, counts, text lengths, and playback timestamps only.

AreaFindingImplementation Consequence
PiP API in in-app browser window.documentPictureInPicture and requestWindow were not exposed in the Codex in-app browser page context. This browser can support DOM research, but not the final activation proof. Use Chrome or Zen control for the real Document PiP click test.
Lyrics button Plex exposes button[data-testid="lyricsButton"][aria-label="Lyrics"]. When lyrics are open, the button includes Plex's selected link class. The existing extension selector remains valid for opening and detecting the lyrics panel.
Timed lyrics toggle Timed mode exposes button[data-testid="toggleTimedLyrics"][aria-label="Timed Lyrics"] and an active class matching AudioVideoLyrics-isTimedLyricsActive-*. The extractor can distinguish timed mode from plain lyric display by checking the timed toggle state.
Lyrics container stack The panel uses classes matching AudioVideoLyrics-container-*, AudioVideoLyrics-content-*, AudioVideoLyrics-innerContent-*, and AudioVideoLyrics-scroller-*. Use prefix/contains selectors, not full hashed class names, and keep fallbacks because Plex may rebuild class hashes.
Line model Timed lyrics are repeated div[class*="AudioVideoLyrics-line-"] rows. Text rows contain a child span; stanza breaks appear as empty line rows; each row can contain a timed icon SVG. A snapshot can preserve lyric order, text rows, and stanza breaks. Use span text for display and empty rows as spacing markers.
Active line signal The live current line adds a class matching AudioVideoLyrics-isLineSelected-*. Its timed icon becomes visible. During playback, selected line indices advanced as audio time advanced. This is the key sync proof: PiP can mirror Plex's active line directly instead of deriving timing from audio playback.
Playback surface Plex creates a hidden audio element with class matching HTMLMedia-mediaElement-*. Player buttons expose stable test IDs such as resumeButton, pauseButton, and lyricsButton. The extractor can observe playback state for diagnostics, but should not depend on audio analysis for lyric sync.
Now-playing surface Plex can expose current track title and artist through navigator.mediaSession.metadata, while the page media element exposes currentTime and duration. The PiP header can show a compact title/artist readout and track progress using session-only browser state. Treat metadata as optional and never persist it.
PiP sizing and placement Document PiP allows requested width and height, and MDN documents that the browser remembers previous size and position by default. The page cannot set the PiP window position. Save only a UI-size preference as {width,height} and rely on the browser for last-position restoration. Do not attempt to store or set screen coordinates.
Remaining DOM gaps Only a timed LyricFind case was mapped in this spike. Plain lyrics, no-lyrics, track change, and page-navigation behavior still need live mapping. Phase 2 is partially complete, not done. Build extractor cases around timed lyrics first, then validate fallback states.
Zen PiP realm result Temporary Zen testing reached the Document PiP path, but the isolated extension content script could not access the returned PiP window document and repeatedly threw SecurityError: Permission denied to access property "document" on cross-origin object, followed by dead-object errors after teardown. Do not hold the PiP window from the isolated extension script. The spike now uses a separate pip-page.js content script in world: "MAIN" so the PiP controller, snapshotter, and renderer run in Plex's page realm.
Zen page-world proof Temporary Zen testing confirmed the page-world Document PiP path works well: timing mirrors Plex's active line and the portrait layout is close to target. Proceed to final testing. Polish pass keeps lyric DOM nodes stable between active-line changes, slows custom scroll animation, reserves active-line marker space to reduce flashing, and adds previous/play-pause/next controls in the PiP header.
Lyrics panel persistence When the PiP window is open, pip-page.js now checks whether Plex's own lyrics panel is visible and clicks Plex's lyrics button if it is closed. This should reopen lyrics automatically after song changes while keeping Plex as the only lyric source. The click is throttled to avoid rapid retry loops during track loading or unavailable-lyrics states.
Auto-open selector regression The first persistence pass used a broad aria-label*="Lyrics" fallback, which also matched Plex's Timed Lyrics toggle and caused timed mode to flip on and off during instrumental sections. Main lyrics-button lookup must prefer data-testid="lyricsButton" and only accept an exact aria-label="Lyrics". Do not use contains-match lyric selectors for auto-click targets.

Market Research

Lyrics have moved from "nice extra" to core music UX. Apple positions real-time lyrics and Sing as an immersive listening feature. Spotify explicitly supports lyrics across free and Premium tiers, with availability varying by rights and device. Plex already has the raw ingredient, but its web experience is tied to the Plex tab and layout.

Market SignalWhat Users Now ExpectOpportunity for This Extension
Apple Music Sing Animated real-time lyrics, translations/pronunciations, duet view, and a TV-friendly big-screen lyric mode. Make the PiP feel premium: large readable current line, dim context lines, smooth movement, and a stage mode that works across the desk.
Spotify Lyrics Lyrics are expected in mainstream music apps, though availability depends on markets, devices, and rights holders. Be honest about availability. If Plex has no lyrics, the extension should say so beautifully instead of looking broken.
Plex LyricFind Plex Pass users can get plain and timed lyrics from LyricFind on supported apps, including Plex Web. The extension can win by presentation, not supply. It mirrors Plex's entitlement into a better multitasking surface.
Desktop multitasking Users want music context while working in other tabs or apps, especially on side screens and ultrawide displays. PiP is a better mental model than a popup because it stays visible, user-positioned, and lightweight.
Licensed lyric ecosystem Lyrics are rights-managed content. Lyric providers exist because display and distribution rights matter. Never become a shadow lyric database. Local-only, session-only display is a strategic constraint, not a compromise.

User Jobs

Work mode: keep the current line readable while coding, browsing, or writing. Side-screen mode: turn a spare monitor into a quiet lyric panel. Sing-along mode: make Plex's timed lyrics feel closer to karaoke without adding microphone, scoring, or third-party data.

Positioning

The best store positioning is "a Picture-in-Picture viewer for lyrics already available in Plex Web." The worst positioning is "get lyrics for Plex" because that sounds like content acquisition and rights handling.

Terms & Store Review

The extension-store story is manageable. The Plex/content-rights story is the sharper edge. The release plan should be conservative until the feature has a working local proof and the exact behavior is easy to describe.

Policy AreaRiskRelease-Safe Posture
Plex terms Plex permits some "Interfacing Software" but restricts modification, derivative use, security circumvention, unauthorized content use, and further distribution of content. Do not call APIs outside normal Plex Web behavior. Do not bypass Plex Pass or region availability. Do not store, export, transmit, or share lyric text. Treat the PiP as an alternate local viewport.
Lyrics rights LyricFind content is licensed and geographically constrained. Lyrics are text content, not neutral metadata. Mirror only what Plex already renders to the authenticated user. Avoid screenshots in store listings that show copyrighted lyric text; use fake placeholder lyrics in screenshots.
Branding Using "Plex" too prominently can imply affiliation; using logos or trade dress increases risk. Use clear unofficial wording: "Lyrics PiP for Plex Web" or "Lyrics Companion for Plex Web", with "not affiliated with Plex" in listing and first-run copy. Avoid Plex logo assets.
Chrome Web Store Chrome requires clear disclosure, narrow permissions, unique value, and no misleading behavior or IP infringement. Single purpose: "show Plex Web lyrics in PiP." Host permissions limited to Plex Web and self-hosted /web/ only if defensible. No remote code. Privacy policy required if handling any user data.
Firefox AMO Mozilla requires no surprises, reviewable code, necessary permissions only, no remote code, and clear data-transmission control. State "local only, no lyric transmission" prominently. Keep code plain/reviewable. Provide reviewer instructions and a Plex test setup note.
Privacy Track titles and lyrics can reveal taste, mood, religion, politics, health context, and household activity. No analytics by default. Do not collect song, artist, server, or lyric text. Store only UI preferences and feature state.
Public release blocker: if any implementation copies full lyrics into extension storage, sync storage, logs, telemetry, screenshots, crash reports, or external services, stop and redesign before publishing.

Store Listing Guardrails

DoAvoid
Say it works only with lyrics already available in Plex Web. Do not say "adds lyrics to Plex" or "unlocks lyrics".
Use placeholder lyric text in screenshots and demos. Do not show copyrighted lyric excerpts in promotional images.
Explain Plex Pass / region / track availability caveats. Do not promise lyrics for every song.
State "not affiliated with or endorsed by Plex" plainly. Do not use Plex logos, official product screenshots with copyrighted content, or store metadata that looks official.

Design Direction

The PiP should feel like a portrait phone-shaped karaoke panel: roughly a third to half of a 1080px-tall display by default, user-resizable, and visually focused almost entirely on lyrics. It should not carry album cards or big metadata chrome; the header can hold only essential playback controls, a tiny now-playing/progress readout, and the on-color close button. Header controls must reserve real layout space so lyrics never scroll behind them.

Design PrincipleExecutionWhy It Matters
Portrait lyric viewport Use a phone-like portrait aspect as the default. Show a slice of the song, not the whole lyric sheet. The user wants a compact always-on-top panel that feels like a lyric screen, not a dashboard.
Karaoke scroll behavior The active line marches down until around the middle of the viewport, then the lyric list scrolls upward; near the end, the active line can march toward the bottom. This follows familiar karaoke behavior and avoids reinventing line timing or motion language.
Plex-adjacent, not Plex-clone Dark surface, warm gold active-line accent, no Plex marks, minimal metadata. It should belong near Plex without implying official ownership.
Lyrics availability is a state Synced lyrics auto-scroll. Plain lyrics remain user-scrollable. No-lyrics and lost-source states use calm plain copy. Rights and metadata variability are normal. The UI should explain them without blame.
Low-noise controls Keep previous/play-pause/next, a small title/progress strip, and an on-color close button in a compact header. Lyrics are the content. Controls and playback context should be available without shouting over the active line.
Accessible separation Use a real header row above the lyric scroller rather than absolutely positioning controls over lyrics. Text remains readable at every scroll position, focus outlines stay visible, and resize reflow never sacrifices lyric contrast.
Accessibility first High contrast, reduced motion, keyboard-reachable controls, no color-only state. Large always-on-top lyrics are an accessibility feature as much as a delight feature.

Visual Ingredients

Near-black panel, soft warm edge, dim surrounding lyrics, clear white active lyric, Plex-gold active marker, tiny track/progress readout, and a small gold close button. No album art unless a later iteration proves it helps.

Motion Rules

Use standard karaoke movement: gentle line emphasis and smooth vertical scroll. No bouncing, flashing, or decorative motion. When reduced motion is requested, update with a simple opacity swap and instant scroll.

Mockups

Design feedback pivot: the earlier mockups had too much extra surface area. The selected direction is a portrait Karaoke Desk: a phone-shaped lyrics viewport with about a quarter of the lyric sheet visible, active-line emphasis, standard karaoke scrolling, user-scrollable plain lyrics, and a small on-color close button.

Selected direction: portrait Karaoke Desk. Keep album art, large metadata panels, source chips, and big progress rails out of the first PiP spike. A narrow header progress hairline is acceptable because it supports playback orientation without turning the PiP into a dashboard.

Final Direction - Portrait Karaoke Desk

StateBehaviorSpike Requirement
Timed lyrics Highlight Plex's selected line, scroll toward the middle, and clamp near the end so the active line can march down. Mirror the AudioVideoLyrics-isLineSelected-* class; do not calculate timing independently.
Plain lyrics Show the full sheet in the same portrait surface and let the user scroll manually. Preserve manual scroll when there is no active line.
No lyrics visible Show a quiet waiting state inside the PiP and keep watching Plex. Never fetch another source or store lyric text.
Design checkpoint: the first implementation uses a 280x500 default PiP request, close to a portrait phone panel. Browsers may clamp the size, and user resizing is acceptable as long as the layout keeps the portrait lyric-reading character.

Architecture

The current extension already has a good spine: popup, MV3 background/session state, content script, Plex page observation, and toast. PiP should extend that spine rather than replacing it.

ModuleResponsibilityImplementation Notes
pip-page.js lyrics extractor Find Plex lyrics container, read visible lines, identify active line, detect plain/timed/no-lyrics states. Runs in the page realm so it can share origin with the Document PiP window. Use structured DOM observation. When PiP is open, throttle-click Plex's lyrics button if the source lyrics panel is closed. Never send lyric text to background.
pip-page.js PiP controller Own the PiP window because Document PiP is tied to the page context. Open from trusted in-page click. Render and update DOM directly inside the PiP window. Close when source tab unloads. Do not access the PiP window from isolated extension code in Zen/Firefox.
pip-page.js PiP playback bridge Expose previous, play/pause, next, compact now-playing, and progress inside the PiP without adding album art or metadata chrome. Use page-world closures to click Plex's own visible player buttons. Read Media Session metadata and media-element time locally. Keep controls icon-only, keyboard-focusable, and local to the current Plex page.
pip-page.js PiP viewport memory Restore the last user-sized PiP viewport where the browser permits it. Persist only {width,height} as a UI preference. Set preferInitialWindowPlacement: false so the browser can remember placement. Position cannot be set by the page.
Injected in-page control Expose "Lyrics PiP" where activation works and where the user understands the action. Keep tiny, Plex-adjacent, and opt-in. The current spike renders a fixed chip near the Plex player controls.
background.js Track feature availability and session state, but not lyric bodies. Store booleans/status only: PiP supported, PiP open, source tab ID, last event. Do not persist lyrics.
popup.js Show PiP availability and guide the user to the activation surface. If direct open fails or is impossible, primary action focuses Plex and reveals the in-page control.
Optional renderer split Separate lyric snapshot creation from rendering. Use a small internal data shape: { track, artist, mode, lines, activeIndex, updatedAt }. Keep it in memory.

Data Boundary

DataAllowedNot Allowed
Lyric text In-memory in the Plex tab and PiP window. Extension storage, sync storage, logs, telemetry, network, clipboard unless user explicitly copies from Plex itself.
Track metadata In-memory display and possibly last-event text like "Lyrics visible". Analytics, crash reports, external services, or persistent listening history.
User preferences Density, font size, reduced motion, last PiP dimensions if browser does not remember. Anything that identifies the Plex server, library contents, or user account.

Step Plan

The order matters: prove browser/API viability, then prove Plex lyric state extraction, then make it beautiful, then decide whether public release is worth the legal/store review effort.

Phase 1 - Browser and activation spike

In a temporary local build, inject a small button into Plex Web and call window.documentPictureInPicture.requestWindow() from the trusted click. Test Zen, Firefox, Chrome, and Edge. Record exact failures and whether classic video PiP is viable. Status: Zen temporary testing proves the page-world pip-page.js path opens and syncs well. Chrome/Edge and fallback behavior remain release-compatibility tests.

Phase 2 - Plex lyrics DOM mapping

Open real Plex Music tracks with timed lyrics, plain lyrics, and no lyrics. Capture selectors, active-line state, mutation patterns, track-change behavior, and whether hidden lyrics remain in the DOM. Update this plan with the findings before building the renderer. Status: timed LyricFind lyrics are mapped; plain/no-lyrics and track-change cases remain.

Phase 3 - In-memory lyric snapshot model

Create a page-local snapshot model for lines, active index, availability state, and timestamps. Add guards so lyric bodies never enter persistent extension storage. Status: first spike snapshots line text, active line, blank rows, and timed/plain/empty mode in the Plex page/PiP runtime only; track title/artist and progress are read only from browser Media Session/media-element state and are not persisted.

Phase 4 - PiP renderer prototype

Build the portrait Karaoke Desk renderer in Document PiP with fake lyrics first, then connect it to the snapshot model. Include plain lyrics and no-lyrics states from the start. Status: renderer is connected directly to the Plex snapshot, keeps lyric nodes stable between active-line changes, auto-reopens Plex's source lyrics panel while PiP is open, and uses fake copy only for empty/waiting states.

Phase 5 - Control surface integration

Add the in-Plex "Open Lyrics PiP" chip and popup guidance. Keep the control opt-in, unobtrusive, and state-aware. Verify it does not fight the existing keep-lyrics-open session model. Status: in-page chip exists; PiP header now includes previous/play-pause/next controls, a compact now-playing readout, and a progress bar that bridge/read from Plex's own page state.

Phase 6 - Accessibility and performance polish

Add density modes, font sizing, reduced motion, keyboard focus, high contrast, resize handling, multi-tab behavior, and observer throttling. Stress test long songs and rapid track changes. Status: first polish pass slows custom scroll animation, reserves active-line marker space, patches active-line classes instead of rebuilding the full lyric DOM, keeps header controls keyboard-focusable, prevents lyrics from scrolling behind the header, and saves/restores the last PiP viewport size.

Phase 7 - Compliance preflight

Audit permissions, data handling, screenshots, store listing copy, naming, trademark posture, privacy policy, and reviewer instructions. Decide Q3 before preparing a public submission.

Phase 8 - Release candidate or private-only package

Package for the chosen browser target. If public release is approved, prepare fake-lyrics screenshots and exact caveat copy. If private-only, document temporary-load and rollback procedures.
